## What It Is

A `docId` is a unique, deterministic, and human-readable identifier for every compiled article within the YAAF Knowledge Base [Source 8]. It serves as the primary key for an article, used for retrieval, cross-referencing, and internal tracking throughout the framework's subsystems.

The `docId` is designed to be stable across recompilations. As long as an article's canonical title and [Entity Type](./entity-type.md) remain the same, its `docId` will not change.

The standard format for a `docId` is `{pluralized-entity-type}/{slug}`, where the slug is a URL-friendly version of the article's canonical title [Source 6]. For example, an article with the title "Attention Mechanism" and entity type "concept" would have the `docId` `concepts/attention-mechanism` [Source 3, 6].

## How It Works in YAAF

The `docId` is central to the entire knowledge base lifecycle, from compilation to runtime access.

### Generation

A `docId` is generated by the [generateDocId](../apis/generate-doc-id.md) utility function during the knowledge base compilation process. This function takes the article's canonical title and [Entity Type](./entity-type.md) as input and produces the standardized identifier string [Source 6].

### Usage in the Compiler

During compilation, the `docId` is used extensively:
*   **Deduplication:** In [Semantic Deduplication](./semantic-deduplication.md), when multiple article plans are merged, the `docId` is used as a deterministic tie-breaker. The plan with the lexicographically smallest `docId` becomes the survivor [Source 2].
*   **Tracking and Progress:** Throughout the synthesis phase, events and results are associated with a `docId` to track the status of each article being created or updated [Source 5]. The compiler can also be configured to skip a set of `docId`s during incremental builds [Source 5].
*   **Indexing:** The [Reverse Citation Index](./reverse-citation-index.md) uses the `docId` as a key to map compiled articles back to the source files they were generated from (`articleToSources[docId]`) [Source 1].
*   **Linting and Post-processing:** When reading compiled articles for processes like linting or the vision pass, the `docId` is used to identify and load each article [Source 3, 7].
*   **Linking:** In [Frontmatter](./frontmatter.md), fields of type `entity_ref` use a `docId` to create explicit relationships between articles. The compiler validates that these `docId`s correspond to existing articles in the knowledge base [Source 4].

### Usage at Runtime

At runtime, the `KBStore` uses the `docId` as the primary means of accessing knowledge:
*   It is the key for retrieving a specific [CompiledDocument](../apis/compiled-document.md) from the store [Source 8].
*   All in-memory metadata structures, such as [DocumentMeta](../apis/document-meta.md) and [KBIndexEntry](../apis/kb-index-entry.md), are keyed or identified by their `docId` [Source 8].
*   Search results returned by a [KBSearchAdapter](./kb-search-adapter.md) include the `docId` for each matching article, allowing the application to retrieve the full content [Source 8].
