## What It Is

[Frontmatter](./frontmatter.md) Validation is a crucial step in the YAAF knowledge base compilation process where the YAML frontmatter generated by a Large Language Model ([LLM](./llm.md)) for a new article is checked for correctness and consistency against a predefined schema [Source 1]. The schema is defined by the knowledge base's [Ontology](./ontology.md) for a given [Entity Type](./entity-type.md).

The primary purpose of this process is to enforce structural integrity in the knowledge base's metadata. It catches common LLM errors such as using incorrect values for enumerated fields, omitting required fields, or formatting data incorrectly. The validation strategy is designed to be strict for required fields to ensure essential metadata is present, while being more lenient for optional fields to avoid blocking the entire compilation process over minor issues [Source 1].

## How It Works in YAAF

The validation and frontmatter generation process is managed by several functions within the [Knowledge Compiler](../subsystems/knowledge-compiler.md)'s synthesizer subsystem [Source 1].

First, the raw markdown output from the LLM, which is prompted to include a YAML frontmatter block, is parsed by the `parseArticleOutput` function. This function separates the frontmatter from the article body and can handle cases where the LLM fails to generate a frontmatter block [Source 1].

The core validation logic resides in the `validateFrontmatter` function. This function compares the parsed frontmatter values against the `FrontmatterSchema` defined in the ontology for the article's specific entity type. It applies coercion and validation rules based on the defined `FieldType` for each field [Source 1]:

*   **string / url**: Values are coerced to strings.
*   **string[] / url[]**: Values are coerced into an array of strings.
*   **number**: String values are parsed into numbers if possible.
*   **boolean**: The strings `'true'` or `'false'` are coerced to boolean values.
*   **date**: Accepted as a string, but a warning is issued if it is not in ISO format.
*   **enum / enum[]**: The value or values are checked against a predefined list of allowed options.
*   **entity_ref / entity_ref[]**: The document ID(s) are checked for existence within the `ConceptRegistry`.

After validation, the `buildCompleteFrontmatter` function assembles the final frontmatter by merging data from multiple sources in a specific order of priority [Source 1]:

1.  **Compiler-suggested frontmatter**: Values suggested by an earlier stage of the compiler (the ExtractorPlan).
2.  **LLM-generated values**: The frontmatter provided by the LLM, after it has been successfully validated.
3.  **[Compiler-injected Metadata](./compiler-injected-metadata.md)**: System-generated metadata that is always added and overwrites any conflicting keys from the other sources.

This final set of compiler-injected metadata ensures structural consistency across all articles and includes fields such as `entity_type`, `stub`, `compiled_at`, `compiled_from`, `compiled_from_quality`, and `confidence` [Source 1].

Finally, the complete and merged frontmatter object is serialized into a standard YAML block (between `---` delimiters) by the `serializeFrontmatter` function, ready to be prepended to the compiled article file [Source 1].
