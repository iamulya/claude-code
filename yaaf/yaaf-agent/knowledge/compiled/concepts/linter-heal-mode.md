## What It Is
Linter Heal Mode is an opt-in, LLM-powered repair mechanism within the YAAF knowledge compiler. It is designed to resolve complex linting issues that require semantic understanding and cannot be addressed through simple string replacement or deterministic rules.

The primary purpose of Heal Mode is to maintain the integrity and quality of the knowledge base by automatically correcting structural and content-related errors. It bridges the gap between identifying a problem (linting) and resolving it (compilation) by leveraging Large Language Models (LLMs) to make informed decisions about article content and relationships.

## How It Works in YAAF
Heal Mode is implemented via the `healLintIssues` function. It processes a `LintReport` generated by the framework's linter and attempts to resolve specific categories of issues. Because the process involves LLM calls, it is an opt-in feature that typically requires an explicit `--heal` flag during the compilation process.

### Supported Issue Types
The mechanism is capable of addressing the following linting codes:
*   **BROKEN_WIKILINK**: The system searches for the best matching article within the registry to update the link or removes the link if no suitable target is found.
*   **STUB_WITH_SOURCES**: If an article is marked as a stub but has associated source material, Heal Mode triggers a re-synthesis to expand the stub into a full encyclopedic article.
*   **LOW_ARTICLE_QUALITY**: The LLM analyzes thin sections of an article and expands them with additional detail based on available context.
*   **ORPHANED_ARTICLE**: The system identifies related articles within the knowledge base and suggests or inserts appropriate crosslinks to improve discoverability.

### Limitations
Heal Mode does not resolve all linting issues. Specifically, it defers the following to human review or discovery phases:
*   **Missing Articles**: If there is no source material available to generate the content, the system cannot create the article.
*   **Contradictory Claims**: Issues where sources provide conflicting information require human intervention to determine the ground truth.

## Configuration
Developers can configure the behavior of the healing process using the `HealOptions` object. Key parameters include `maxCalls` (which defaults to 20 to control costs and execution time) and `dryRun` (which allows developers to preview changes without modifying the underlying files).

```typescript
import { healLintIssues } from './knowledge/compiler/heal.js';

const llm = makeKBLLMClient();
const lintReport = await compiler.lint();

const healResult = await healLintIssues(
  llm,
  lintReport,
  compiledDir,
  registry,
  {
    maxCalls: 50,
    dryRun: false
  }
);

console.log(`Healed ${healResult.healed} issues using ${healResult.llmCalls} LLM calls.`);
```

The `HealResult` provides a detailed breakdown of the operation, including the duration, the number of issues skipped, and a per-issue log of actions taken via the `HealDetail` type.
