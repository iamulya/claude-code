## What It Is
Tool Call Authorization is a security mechanism in YAAF designed to intercept and validate tool execution requests generated by a Large Language Model (LLM). Because LLMs can occasionally produce unintended or malicious tool calls—either through hallucinations or prompt injection—YAAF implements a policy-based layer to ensure that every action remains within defined safety boundaries.

This concept solves the "agency problem" by providing developers with fine-grained control over which tools an agent can execute, which arguments are permissible, and which actions require explicit human-in-the-loop confirmation.

## How It Works in YAAF
The authorization system is centered around the `PermissionPolicy` class. When an agent attempts to invoke a tool, the framework passes the request through this policy to determine a `PermissionOutcome`.

### Decision Outcomes
A policy evaluates a tool call into one of three primary actions:
1.  **Allow**: The tool call is executed immediately.
2.  **Deny**: The tool call is blocked, and an error (or a specific denial reason) is returned to the agent.
3.  **Escalate/Require Approval**: The tool call is paused until a human or an external system grants permission via an `ApprovalHandler`.

### Pattern Matching
YAAF uses glob-style pattern matching to associate rules with tool names and their arguments:
*   `tool_name`: Matches any call to a specific tool.
*   `tool_name(*)`: Matches any call to a specific tool (equivalent to the above).
*   `tool_name(pattern)`: Matches if the serialized JSON representation of the tool arguments includes the specified pattern.
*   `prefix_*`: Matches any tool starting with the specified prefix (e.g., `read_*` matches `read_file` and `read_db`).

### Safety Predicates
The framework includes built-in utilities for content-aware blocking. The `isDangerousCommand()` function creates a predicate that scans tool arguments against `DANGEROUS_PATTERNS`. These patterns include:
*   Destructive filesystem operations (e.g., `rm -rf /`, `mkfs.*`).
*   Privilege escalation (e.g., `sudo`).
*   Insecure permission changes (e.g., `chmod 777`).
*   Remote script execution (e.g., `curl ... | bash`).
*   Fork bombs and raw disk writes.

## Configuration
Developers configure authorization by attaching a `PermissionPolicy` to the `Agent` instance.

### Basic Policy
```ts
const agent = new Agent({
  systemPrompt: '...',
  permissions: new PermissionPolicy()
    .allow('search_*')
    .allow('check_weather')
    .requireApproval('book_trip', 'Booking requires your confirmation')
    .deny('delete_*', 'Deletion is not permitted')
    .onRequest(async (toolName, args, reason) => {
      // Custom logic for manual approval
      const answer = await readline.question(`Allow ${toolName}? [y/N] `);
      return answer.toLowerCase() === 'y';
    }),
});
```

### Secure CLI Policy
For command-line agents, YAAF provides a pre-configured `secureCLIPolicy()` that blocks dangerous shell patterns and requires approval for execution commands while allowing read-only operations.

```ts
const policy = secureCLIPolicy()
  .allow('read_*')  // override: allow all reads without prompt
  .onRequest(cliApproval()); // Use built-in CLI prompt handler
```

### Content-Aware Denials
Policies can use the `when` predicate to block specific argument values, such as dangerous shell commands.

```ts
policy.deny('exec_command', 'Dangerous command detected', {
  when: isDangerousCommand(),
});
```
