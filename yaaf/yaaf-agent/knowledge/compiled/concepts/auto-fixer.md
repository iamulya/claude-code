## What It Is

Auto-Fixing is a capability within the YAAF knowledge base [Linter](./linter.md) that automatically corrects certain categories of detected issues in compiled article files [Source 1]. Its purpose is to reduce the manual effort required to maintain the structural and semantic integrity of the knowledge base by programmatically resolving common, unambiguous errors.

The Auto-Fixer component is designed to be conservative and non-destructive, only performing additions or safe rewrites of content [Source 1].

The specific issues that are auto-fixable include [Source 1]:
*   **[NON_CANONICAL_WIKILINK](./non-canonical-wikilink.md)**: Rewrites a [wikilink](./wikilinks.md) that uses an alias to its canonical title (e.g., `[[alias]]` becomes `[[canonical title]]`).
*   **[UNLINKED_MENTION](./unlinked-mention.md)**: Adds a wikilink to the first occurrence of a term that matches a known entity's title but is not currently linked.
*   **[MISSING_REQUIRED_FIELD](./missing-required-field.md)**: Injects a required field with a default value into an article's [Frontmatter](./frontmatter.md) if it is absent.

## How It Works in YAAF

The Auto-Fixing logic is implemented in the `Auto-Fixer` component and exposed through the [applyFixes](../apis/apply-fixes.md) function. This function takes a list of `LintIssue` objects, generated by the [Linter](./linter.md), and the path to the compiled articles directory as input [Source 1].

The process operates with the following conservative principles [Source 1]:
1.  **Incremental Application**: Fixes are applied one at a time to build up the final changes for a file.
2.  **Batched File Writes**: All changes for a given file are grouped together, ensuring that only a single write operation is performed per file.
3.  **Stale Issue Skipping**: If a fix for one issue modifies the text where another issue was located, the fixer will skip the second issue if its target text can no longer be found. This prevents conflicts from multiple fixes targeting the same text region.
4.  **Non-Destructive**: The fixer never removes content. Its operations are limited to adding content (like frontmatter fields) or rewriting [Wikilinks](./wikilinks.md).

The [applyFixes](../apis/apply-fixes.md) function also supports a `dryRun` mode, which calculates the proposed changes without writing them to disk, allowing for a preview of the fixes [Source 1].

## Configuration

The primary way to control the Auto-Fixing process is by invoking the [applyFixes](../apis/apply-fixes.md) function with the `dryRun` parameter.

```typescript
import { applyFixes } from "./fixer.js";
import type { LintIssue } from "./types.js";

// Assume 'issues' is an array of LintIssue objects from the linter
const issues: LintIssue[] = [ /* ... */ ];
const compiledDir = "./compiled/knowledge";

// To apply fixes directly to files:
const result = await applyFixes(issues, compiledDir, false);
console.log(`Applied ${result.fixedIssues.length} fixes.`);

// To preview fixes without modifying files:
const dryRunResult = await applyFixes(issues, compiledDir, true);
console.log("Dry run results:", dryRunResult);
```
