## What It Is

Wikilink Resolution is a compile-time transformation within the YAAF [Knowledge Base Compiler](../subsystems/knowledge-base-compiler.md). It is responsible for converting simple, human-readable `wikilink` notations within article markdown into standard, relative markdown links that point to other files in the knowledge base [Source 1].

This process is a key post-synthesis step, occurring after an article's content is generated by an [LLM](./llm.md) but before the final [Linting](./linting.md) checks are performed. It allows authors (both human and AI) to reference other concepts by name without needing to know or hardcode their specific file paths, making the knowledge base more maintainable and robust against file reorganizations [Source 1].

## How It Works in YAAF

The core logic for this process is encapsulated in the `resolveWikilinks` function, which is part of the larger `postProcessCompiledArticles` step in the knowledge base compiler [Source 1].

The resolution mechanism operates as follows:

1.  **Parsing**: The compiler scans the markdown content of an article for text matching the wikilink syntax. It supports two forms [Source 1]:
    *   `Target Concept`: A direct link where the link text is the same as the target concept's title.
    *   `custom display text`: A link with custom display text, similar to standard markdown links.

2.  **Registry Lookup**: For each found wikilink, the compiler queries the `ConceptRegistry`. This registry is a central index that maps concept titles (like "Target Concept") to their metadata, including their unique `docId` and [Entity Type](./entity-type.md) [Source 1].

3.  **Path Generation**: If a matching entry is found in the registry, the compiler uses the target's `docId` and entity type to construct a relative file path from the current article to the target article. For example, `Attention Mechanism` might be resolved to `[Attention Mechanism](../concepts/attention-mechanism.md)` [Source 1].

4.  **Handling Unresolved Links**: If a wikilink's target cannot be found in the `ConceptRegistry`, the link is intentionally left in its original `Target` format. This allows a subsequent linting step to identify and report these "broken" internal links [Source 1].

5.  **Dependency Tracking**: As part of the process, the compiler records which articles link to which other articles. This dependency map (`wikilinkDeps`) is used by the differential build engine to intelligently recompile only the necessary files [when](../apis/when.md) a concept is updated. The compiler also tracks which articles still contain unresolved links (`unresolvedDocIds`) to target them for future updates when new concepts are added to the registry [Source 1].

## Configuration

The Wikilink Resolution step can be configured or disabled via the `PostProcessOptions` object passed to the compiler. The `resolveLinks` property controls whether this transformation is applied [Source 1].

```typescript
// Example: Disabling wikilink resolution during post-processing
export type PostProcessOptions = {
  /** Token budget per article. Default: 15000 */
  tokenBudget?: number;
  /** Whether to resolve wikilinks. Default: true */
  resolveLinks?: boolean;
  /** Whether to segment oversized articles. Default: true */
  segmentArticles?: boolean;
};

// Usage in the compiler
await postProcessCompiledArticles(compiledDir, registry, {
  resolveLinks: false, // This would skip the resolution step
});
```

By default, `resolveLinks` is set to `true` [Source 1].
