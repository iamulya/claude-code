## What It Is
Wikilink Resolution is a compile-time transformation in YAAF that converts internal knowledge base references—written in the `[[Target]]` or `[[Target|Display Text]]` syntax—into standard Markdown relative links. 

This mechanism serves three primary purposes:
1.  **Authoring Efficiency**: It allows authors and the Knowledge Synthesizer to link between articles without needing to know the final file system structure or relative paths.
2.  **Link Integrity**: By resolving links against a centralized registry, the framework ensures that internal links point to valid documents.
3.  **Validation**: Unresolved wikilinks remain in their raw format, allowing the framework's linter to identify and flag missing articles or typos in references.

## How It Works in YAAF
Wikilink resolution is implemented as a post-synthesis processing step (B1) within the `postprocess.ts` module. It occurs after an article has been generated by the LLM but before the final linting phase.

### The Resolution Process
The resolution logic is handled by the `resolveWikilinks` function, which performs the following steps:
1.  **Registry Lookup**: For every wikilink found in the markdown source, the compiler queries the `ConceptRegistry` using the `findByWikilink` function.
2.  **Matching Priority**: The registry attempts to find a match based on a specific hierarchy:
    *   **Exact docId match**: e.g., `concepts/attention-mechanism`.
    *   **Canonical title match**: A case-insensitive match against the `title` field in an article's frontmatter.
    *   **Alias match**: A case-insensitive match against any defined aliases for an article.
3.  **Path Computation**: If a match is found, the compiler calculates the relative path from the current document's `docId` to the target document's location.
4.  **Transformation**: The syntax is converted. For example:
    *   `[[Attention Mechanism]]` becomes `[Attention Mechanism](../concepts/attention-mechanism.md)`.
    *   `[[Attention|custom text]]` becomes `[custom text](../concepts/attention-mechanism.md)`.

### The Concept Registry
The resolution process relies entirely on the `ConceptRegistry`, a live, in-memory index of every entity in the compiled knowledge base. The registry is built by scanning the `compiled/` directory for Markdown files and extracting their YAML frontmatter. It provides a reverse alias lookup (lowercase alias → docId) to facilitate fast mapping during the resolution phase.

## Configuration
Wikilink resolution is part of the `PostProcessOptions` and is typically enabled by default during the compilation pipeline. Developers can control this behavior when calling `postProcessCompiledArticles`.

```typescript
export type PostProcessOptions = {
  /** Whether to resolve wikilinks. Default: true */
  resolveLinks?: boolean
  /** Token budget per article. Default: 15000 */
  tokenBudget?: number
  /** Whether to segment oversized articles. Default: true */
  segmentArticles?: boolean
}
```

If `resolveLinks` is set to `false`, the compiler will skip the transformation, leaving all `[[wikilinks]]` in their raw state.
