## What It Is

A WorkerAgent is a specialized agent role within YAAF's "[Coordinator Mode](../subsystems/coordinator-mode.md)," a first-class multi-agent pattern for task decomposition and execution [Source 1]. In this pattern, a central [CoordinatorAgent](./coordinator-agent.md) breaks down a complex problem and delegates specific sub-tasks to one or more WorkerAgents.

Each WorkerAgent is typically equipped with a distinct set of [Tools](../subsystems/tools.md) and capabilities tailored to a specific function, such as research, code implementation, or data analysis [Source 1]. The primary responsibility of a WorkerAgent is to execute its assigned task and report the outcome back to the coordinator in a structured format. This division of labor allows for more complex problems to be solved by combining the outputs of multiple specialized agents.

## How It Works in YAAF

A WorkerAgent operates under the direction of a CoordinatorAgent. The coordinator is configured with a list of available workers, each defined by its ID, a description of its capabilities, and the tools it can use [Source 1]. The CoordinatorAgent's [System Prompt](./system-prompt.md), generated by the `CoordinatorPromptBuilder`, includes information about these workers, enabling the [LLM](./llm.md) to delegate tasks effectively [Source 1].

Upon completing or failing a task, a WorkerAgent automatically reports its status and results back to the CoordinatorAgent. This communication uses a structured data payload called a `[[[[[[[[TaskNotification]]]]]]]]` [Source 1]. The `TaskNotification` object contains the task ID, a status (e.g., `completed`, `failed`), a summary of the outcome, and optional fields for detailed results and resource usage metrics [Source 1].

The CoordinatorAgent receives these notifications, often through events like `worker:completed`, and synthesizes the results from its workers to formulate a final response or decide on the next steps [Source 1]. The `buildWorkerResult` utility function can be used to format a worker's output into a message that can be injected back into the coordinator's context for synthesis [Source 1].

## Configuration

WorkerAgents are not instantiated directly but are defined as part of the configuration for a CoordinatorAgent. A developer provides an array of worker definitions [when](../apis/when.md) creating the coordinator. Each definition specifies the worker's `id`, `description`, and the `tools` it has access to.

```typescript
const coordinator = createCoordinator({
  model,
  workers: [
    { id: 'researcher', tools: [searchTool, readTool], description: 'Research agent' },
    { id: 'implementer', tools: [editTool, writeTool], description: 'Implementation agent' },
  ],
});

coordinator.on('worker:completed', (notif) => {
  console.log(`Worker ${notif.taskId} finished: ${notif.summary}`);
});

const result = await coordinator.run('Fix the auth bug in login.ts');
```
