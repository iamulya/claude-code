## Purpose

The Security System in YAAF addresses two critical security concerns in LLM-powered agent applications: supply chain integrity and output validation [Source 1, Source 2].

1.  **Integrity Verification**: It provides a mechanism to verify the integrity and trustworthiness of external components like plugins and Multi-Agent Communication Protocol (MCP) servers before they are loaded or used. This mitigates the risk of running malicious or compromised code by enforcing checks such as SHA-256 hash validation for plugins and allow/block lists for tools exposed by remote servers [Source 2].

2.  **Output Validation**: It offers tools to validate and constrain the outputs generated by Large Language Models (LLMs). This prevents downstream application logic from processing malformed, unexpected, or potentially harmful structured data (e.g., JSON). It ensures that LLM outputs conform to a predefined schema, protecting against errors and injection-style vulnerabilities [Source 1].

## Architecture

The Security System is composed of two primary, independent components: `TrustPolicy` and `StructuredOutputValidator`.

### TrustPolicy

The `TrustPolicy` class is the core of the integrity verification mechanism. It is configured with a manifest that defines trust rules for plugins and MCP servers. Its key architectural features include:

*   **Plugin Verification**: For each plugin, the policy can check its content against an expected SHA-256 hash or verify that its version satisfies a specified semantic versioning range [Source 2].
*   **MCP Server Tool Filtering**: For each MCP server, the policy can enforce an `allowedTools` list or a `blockTools` list, restricting which remote tools the agent is permitted to use [Source 2].
*   **Operational Modes**: The policy can operate in two modes:
    *   `strict`: Fails and prevents the loading of any component that does not pass verification (default) [Source 2].
    *   `warn`: Logs a warning for verification failures but allows the component to be loaded [Source 2].
*   **Unknown Policy**: It defines how to handle plugins or servers that are not explicitly listed in the trust manifest, either by allowing (`allow`) or denying (`deny`) them [Source 2].
*   **Audit Logging**: Every verification attempt generates a `TrustVerificationEvent`, which can be captured via an `onVerification` callback for detailed auditing [Source 2].

### StructuredOutputValidator

The [StructuredOutputValidator](../apis/structured-output-validator.md) class is responsible for schema-based validation of LLM outputs. It is configured with a set of rules to enforce on the data.

*   **Rule-Based Validation**: The validator checks structured data (primarily JSON) against a list of `FieldRule` definitions. These rules can specify field types (string, number, url, etc.), required status, allowed enum values, numeric ranges, string length, and regex patterns [Source 1].
*   **Violation Handling**: When a validation rule is violated, the validator can be configured to take one of three actions:
    *   `warn`: Log the violation and pass the original data through (default) [Source 1].
    *   `strip`: Remove the invalid fields from the object [Source 1].
    *   `reject`: Discard the LLM's output and replace it with an error message [Source 1].
*   **URL Sanitization**: Beyond JSON validation, it can be configured to scan all output text for URLs, checking them against an `allowedDomains` list to prevent access to untrusted domains [Source 1].
*   **Event-Based Reporting**: Each validation check produces an `OutputValidationEvent`, which provides details on the outcome, any violations found, and the action taken. This can be captured via an `onValidation` callback [Source 1].

## Integration Points

*   **[Plugin System](./plugin-system.md)**: The Plugin System would interact with `TrustPolicy` during its loading phase to verify the integrity of each plugin before it is initialized and made available to the agent.
*   **[Agent Collaboration](./agent-collaboration.md)**: The Inter-Agent Communication subsystem would use `TrustPolicy` to filter the list of tools advertised by a remote MCP server, ensuring the agent only sees and uses permitted tools.
*   **[Agent Core](./agent-core.md)**: The core agent loop or response processing logic would use the [StructuredOutputValidator](../apis/structured-output-validator.md) immediately after receiving a structured output from an LLM. This ensures the data is safe and well-formed before it is passed to tools, memory, or other internal systems.

## Key APIs

### TrustPolicy
*   `TrustPolicy`: The main class for defining and enforcing integrity rules [Source 2].
*   `TrustPolicyConfig`: The configuration object passed to the `TrustPolicy` constructor, defining rules for plugins and MCP servers, as well as the operational mode [Source 2].
*   `PluginTrust`: A configuration object that specifies the expected `sha256` hash, `version` constraint, or `trusted` status for a single plugin [Source 2].
*   `McpServerTrust`: A configuration object that specifies `allowedTools`, `blockTools`, or `trusted` status for an MCP server [Source 2].
*   `trustPolicy()`: A factory function for creating a `TrustPolicy` instance with sensible defaults [Source 2].

### StructuredOutputValidator
*   [StructuredOutputValidator](../apis/structured-output-validator.md): The class that performs validation on LLM outputs [Source 1].
*   `OutputValidatorConfig`: The configuration object for the validator, specifying field rules, violation actions, and URL validation settings [Source 1].
*   `FieldRule`: Defines a single validation rule for a field, including its name, type, and constraints like `required`, `allowedValues`, `min`/`max`, and `pattern` [Source 1].
*   `OutputValidationEvent`: An object emitted during validation that details the outcome, including whether the output was valid, what violations occurred, and what action was taken [Source 1].

## Configuration

### TrustPolicy Configuration

A `TrustPolicy` is configured by providing a `TrustPolicyConfig` object. This object acts as a security manifest for the agent.

```typescript
import { TrustPolicy } from 'yaaf';

const trust = new TrustPolicy({
  mode: 'strict',
  unknownPolicy: 'deny',
  plugins: {
    'my-plugin': { sha256: 'abc123...' },
    'another-plugin': { version: '^2.3.0' },
  },
  mcpServers: {
    'github': { allowedTools: ['search_repos', 'get_issue'] },
    'internal-api': { trusted: true },
  },
});
```
This example configures a strict policy that denies unknown components, verifies `my-plugin` against a hash, ensures `another-plugin` is version 2.3.x, restricts the `github` server to two specific tools, and fully trusts the `internal-api` server [Source 2].

### StructuredOutputValidator Configuration

The [StructuredOutputValidator](../apis/structured-output-validator.md) is configured with an `OutputValidatorConfig` object that defines the validation rules.

```typescript
import { StructuredOutputValidator } from 'yaaf';

const validator = new StructuredOutputValidator({
  onViolation: 'reject',
  validateUrls: true,
  allowedDomains: ['example.com'],
  rules: [
    { field: 'user.id', type: 'number', required: true, min: 1 },
    { field: 'user.role', type: 'enum', allowedValues: ['admin', 'user'] },
    { field: 'details.website', type: 'url', required: false },
  ],
});
```
This configuration will reject any output that doesn't conform to the rules. It requires a numeric `user.id` greater than 0, ensures `user.role` is one of two allowed values, and validates that any URLs present are from the `example.com` domain [Source 1].

## Extension Points

The primary way to extend the Security System's behavior is through its event-based callbacks:

*   **`onVerification`**: A function in `TrustPolicyConfig` that receives a `TrustVerificationEvent` every time a plugin or MCP server is checked. This can be used for custom logging, sending metrics to a monitoring service, or triggering alerts on security failures [Source 2].
*   **`onValidation`**: A function in `OutputValidatorConfig` that receives an `OutputValidationEvent` for each LLM output that is validated. This allows developers to log detailed information about validation failures or successes for debugging and analysis [Source 1].
