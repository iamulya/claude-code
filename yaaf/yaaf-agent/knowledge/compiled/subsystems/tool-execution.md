## Purpose
The Tool Execution Subsystem is responsible for the lifecycle of tool calls generated by a Large Language Model (LLM). It specifically addresses the challenges of executing tools as they are parsed from a streaming response, ensuring that results are returned to the LLM in the correct sequence while optimizing performance through parallel execution where safe.

The subsystem acts as a bridge between the raw tool calls identified by the agent runner and the actual implementation of those tools, enforcing security boundaries and handling execution failures.

## Architecture
The core of this subsystem is the `StreamingToolExecutor`. It manages tool execution through several specialized mechanisms:

### Concurrency Management
The executor distinguishes between tools based on their safety profiles:
*   **Parallel Execution**: Tools marked with `isConcurrencySafe = true` are executed simultaneously to reduce total latency.
*   **Exclusive Execution**: Tools that are not concurrency-safe are granted exclusive access, ensuring no other tools run while they are active.

### Ordering and Synchronization
Despite tools potentially finishing out of order due to varying execution times or parallel processing, the subsystem guarantees that results are yielded in the order they were requested by the LLM. This maintains the integrity of the conversation history.

### Error Handling and Sibling Abort
The subsystem implements a "sibling abort" policy. If a tool within a parallel batch encounters an error, the executor can cancel its siblings to prevent inconsistent states or wasted computation.

### Progress Reporting
The architecture supports incremental updates, allowing tools to yield progress messages during long-running operations before the final result is produced.

## Integration Points
The Tool Execution Subsystem interacts with several other components of the YAAF framework:
*   **Permissions and IAM**: It utilizes `PermissionPolicy`, `AccessPolicy`, and `UserContext` to verify if a tool call is authorized within the current data scope.
*   **Sandboxing**: It integrates with the `Sandbox` component to provide isolated environments for tool execution.
*   **Hooks**: It triggers lifecycle events via the `Hooks` system, allowing for observability and external intervention.
*   **Agent Runner**: It receives `ToolCall` objects from the agent's streaming logic and returns `ToolExecutionResult` objects.

## Key APIs

### StreamingToolExecutor
The primary class responsible for orchestrating tool calls. It handles the lookup of tools via `findToolByName` and manages the execution state.

### ToolExecutionResult
A standardized structure representing the outcome of a tool invocation.
```typescript
export type ToolExecutionResult = {
  toolCallId: string
  name: string
  content: string
  error?: boolean
  durationMs: number
}
```

## Extension Points
Developers can influence the behavior of the Tool Execution Subsystem through:
*   **Tool Configuration**: Setting the `isConcurrencySafe` flag on individual tool definitions to enable or disable parallel execution.
*   **Hooks**: Registering callbacks to monitor tool execution, log durations, or modify results before they are returned to the agent.
*   **Custom Sandboxes**: Providing specialized `Sandbox` implementations to control the environment in which tools operate.
