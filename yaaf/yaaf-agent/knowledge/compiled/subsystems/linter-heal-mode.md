## Purpose

The [Linter](../concepts/linter.md) [Heal](../concepts/heal.md) Mode is an [LLM](../concepts/llm.md)-powered subsystem within the YAAF [Knowledge Compiler](./knowledge-compiler.md) designed to automatically repair complex lint issues that cannot be resolved with simple text replacement [Source 1]. It is an opt-in feature, typically activated by a `--heal` command-line flag during the compilation process [Source 1].

This subsystem addresses the following specific categories of documentation defects [Source 1]:

*   **`BROKEN_WIKILINK`**: Analyzes broken internal links and attempts to find the best-matching valid article to link to, or removes the link if no suitable target is found.
*   **`STUB_WITH_SOURCES`**: Triggers a re-synthesis process, using an LLM to expand a stub article into a full-fledged article based on its associated source materials.
*   **`LOW_ARTICLE_QUALITY`**: Instructs an LLM to elaborate on thin or underdeveloped sections within an article to improve detail and quality.
*   **`ORPHANED_ARTICLE`**: Identifies related articles within the knowledge base and suggests or inserts cross-links to integrate the orphaned content.

The subsystem is explicitly designed *not* to handle certain issues, which are deferred to other processes. These include fixing missing articles that have no source material and resolving contradictory claims between articles, which require human review [Source 1].

## Architecture

The core of the Linter Heal Mode is the `healLintIssues` function, which orchestrates the repair process [Source 1]. This function operates on a `LintReport` generated by the linter subsystem. For each healable issue in the report, it uses a provided [LLM Client](../concepts/llm-client.md) (`LLMCallFn`) to generate a proposed fix.

The subsystem relies on several key inputs to perform its function [Source 1]:
*   An **LLM client** (`LLMCallFn`) to provide the generative AI capabilities for rewriting content and fixing links.
*   A **`LintReport`** containing the list of issues to be addressed.
*   The path to the **compiled directory**, where it reads the source files and writes the healed versions.
*   A **`ConceptRegistry`**, which provides an [Ontology](../concepts/ontology.md) of the knowledge base. This is likely used to validate entities and find related articles [when](../apis/when.md) fixing `BROKEN_WIKILINK` and `ORPHANED_ARTICLE` issues.

File modifications are performed using an `atomicWriteFile` utility, ensuring that file writes are safe and complete without leaving files in a corrupted state [Source 1].

## Integration Points

The Linter Heal Mode is tightly integrated with the YAAF knowledge compiler's build pipeline.

*   **Linter Subsystem**: It acts as a consumer of the `LintReport` produced by the linter. It is designed to run immediately after the [Linting](../concepts/linting.md) stage.
*   **LLM Client**: It depends on an `LLMCallFn` implementation, making it agnostic to the specific LLM provider being used.
*   **Ontology Subsystem**: It uses the `ConceptRegistry` to gain semantic understanding of the knowledge base's structure and content, which is crucial for intelligent repairs [Source 1].
*   **File System**: It reads from and writes to the compiled output directory, directly modifying the generated documentation files [Source 1].

## Key APIs

The primary public API for this subsystem is the `healLintIssues` function and its associated data structures.

### `healLintIssues()`

This asynchronous function is the main entry point for running the healing process [Source 1].

```typescript
export async function healLintIssues(
  llm: LLMCallFn,
  report: LintReport,
  compiledDir: string,
  registry: ConceptRegistry,
  options: HealOptions = { /* ... */ }
): Promise<HealResult>
```

*   **Parameters**:
    *   `llm`: An LLM client function for making API calls.
    *   `report`: The lint report containing issues to fix.
    *   `compiledDir`: The directory containing the compiled articles.
    *   `registry`: The knowledge base's [Concept Registry](./concept-registry.md).
    *   `options`: Configuration for the heal run, such as `dryRun` or `maxCalls`.
*   **Returns**: A `Promise` that resolves to a `HealResult` object, summarizing the outcome of the operation.

### `HealResult`

This type represents the summary of a completed heal run [Source 1].

```typescript
export type HealResult = {
  healed: number;      // Number of issues successfully healed
  skipped: number;     // Number of issues skipped
  llmCalls: number;    // Number of LLM calls made
  details: HealDetail[]; // Per-issue details
  durationMs: number;  // Total elapsed time (ms)
};
```

## Configuration

The behavior of the Linter Heal Mode can be controlled via the `HealOptions` object passed to the `healLintIssues` function [Source 1].

```typescript
export type HealOptions = {
  /** Maximum LLM calls per heal run. Default: 20 */
  maxCalls?: number;
  /** Only report what would be healed — don't write changes. Default: false */
  dryRun?: boolean;
  /** Progress callback */
  onProgress?: (event: HealProgressEvent) => void;
};
```

*   `maxCalls`: A safety limit to prevent excessive LLM usage and cost. The default is 20 calls per run.
*   `dryRun`: When set to `true`, the subsystem will report the changes it would make without actually writing them to the file system. This is useful for previewing fixes.
*   `onProgress`: A callback function that receives `HealProgressEvent` objects, allowing callers to monitor the status of the heal process in real-time.
