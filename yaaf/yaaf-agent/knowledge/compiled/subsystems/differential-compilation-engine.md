## Purpose

The Differential Compilation Engine is a subsystem of the YAAF [Knowledge Base Compiler](./knowledge-base-compiler.md) designed to optimize the build process by avoiding unnecessary work [Source 1]. While other incremental mechanisms may operate at the raw file ingestion level, they do not prevent all articles from being proposed for synthesis. This engine implements true article-level differential compilation, ensuring that only articles whose source files have changed are re-synthesized by the [LLM](../concepts/llm.md) [Source 1].

Its primary goals are:
*   To identify articles that are "clean" (unchanged) and can be skipped during the synthesis phase, reducing LLM calls.
*   To identify articles that are "stale" (their source material has changed) and must be recompiled.
*   To identify articles that are "orphaned" (their source files have been deleted) and should be removed from the compiled output.

## Architecture

The engine's logic is encapsulated within the `DifferentialEngine` class. Its operation relies on two key data sources and a comparison algorithm [Source 1].

1.  **Source Hash Manifest**: The engine maintains a manifest file named `.kb-source-hashes.json`. This file stores a map of every raw source file path to its SHA-256 hash from the last successful compilation.

2.  **Compiled Article Scan**: The engine scans the `frontmatter` of all existing compiled markdown articles. It specifically reads the `compiled_from:` field to build a reverse map, linking each compiled article back to the set of source files that contributed to its content.

The diff algorithm proceeds as follows [Source 1]:
1.  It computes the SHA-256 hash for all current raw source files.
2.  It compares these new hashes against the stored hashes in the manifest to create a set of changed, added, or deleted source files.
3.  An article is marked as **STALE** if any of its contributing source files (as determined by the `compiled_from:` field) are in the set of changed files.
4.  An article is marked as **ORPHAN** if all of its contributing source files have been deleted.
5.  Any article that is not stale or orphaned is considered **CLEAN**.

The final output is a compilation plan that categorizes all articles into one of these three states.

## Integration Points

The primary integration point for this subsystem is with the main compiler orchestration logic. The compilation plan generated by the `DifferentialEngine` directly informs the synthesizer. The list of clean document IDs (`cleanDocIds`) is passed to the `CompileOptions.skipDocIds` parameter, which instructs the synthesizer to skip the [LLM Synthesis](../concepts/llm-synthesis.md) step entirely for those articles [Source 1]. The lists of stale and orphan IDs are used by the compiler to manage recompilation and file deletion tasks.

## Key APIs

The public API is exposed through the `DifferentialEngine` class [Source 1].

*   `DifferentialEngine.create(kbDir, rawDir, compiledDir)`: A static factory method that initializes a new engine instance with the necessary directory paths.
*   `async computePlan()`: Computes the differential plan by executing the hashing and comparison algorithm. It returns an object containing three arrays of document IDs: `staleDocIds`, `cleanDocIds`, and `orphanDocIds`.
*   `async save()`: Persists the newly computed source file hashes to the `.kb-source-hashes.json` manifest file on disk, preparing for the next compilation run.

A typical usage pattern is:
```typescript
const diff = await DifferentialEngine.create(kbDir, rawDir, compiledDir);
const plan = await diff.computePlan();
// plan.staleDocIds → recompile these
// plan.cleanDocIds → skip these (no LLM call)
// plan.orphanDocIds → delete these compiled articles
await diff.save(); // persist the new source-hash manifest
```
