## Purpose

The Concept Extractor is the "planning" subsystem within the YAAF Knowledge Base (KB) compilation pipeline [Source 1]. It serves as the bridge between the [Ingester](../apis/ingester.md), which provides raw content, and the [Knowledge Synthesizer](./knowledge-synthesizer.md), which generates the final articles. Its primary responsibility is to analyze ingested source files and produce a structured `CompilationPlan` that dictates what articles to create or update [Source 1].

The pipeline flow is as follows [Source 1]:
`Ingester → [IngestedContent] → ConceptExtractor → [CompilationPlan] → KnowledgeSynthesizer`

## Architecture

The extractor processes each source file through a three-pass architecture to analyze content and form a plan [Source 1].

### Pass 1: Static Analysis
This initial pass operates without using an [LLM](../concepts/llm.md), making it instantaneous. It performs several preparatory steps:
- **[Vocabulary](../concepts/vocabulary.md) Scan**: Scans the source text for mentions of known KB entities by using an alias index.
- **Registry Lookup**: Checks the `ConceptRegistry` to determine which of the mentioned entities already have existing articles.
- **Directory Hint**: Infers a potential [Entity Type](../concepts/entity-type.md) based on the source file's path, using established conventions.
- **[Token Estimation](../concepts/token-estimation.md)**: Calculates the token count of the source content to decide how much of it can be included in the subsequent LLM prompt.

### Pass 2: LLM Classification
This pass utilizes a fast and cost-effective LLM (referred to as the `extractionModel`) to classify and extract key information from the source content:
- **Classification**: Determines the source's entity type and its canonical title.
- **Action Planning**: Decides whether the source corresponds to a new article to be created or an existing one that needs to be updated.
- **Concept [Discovery](../concepts/discovery.md)**: Identifies and extracts candidate concepts mentioned in the text that are not yet part of the KB.
- **[Frontmatter](../concepts/frontmatter.md) Suggestion**: Proposes values for the article's frontmatter based on the source content.
- **Relationship Mapping**: Determines the article's relationships to other known entities in the KB.

### Pass 3: Post-processing
The final pass operates without an LLM to refine and validate the plan generated by the previous passes:
- **ID Generation**: Computes a deterministic `docId` for the article based on the LLM's output.
- **Source Grouping**: Merges multiple source files that have been identified as targeting the same canonical entity.
- **Validation**: Checks the LLM-suggested entity types against the official `KB[[[[[[[[Ontology]]]]]]]]`.
- **Review Flagging**: Marks plans with low confidence scores for subsequent human review.

## Integration Points

The Concept Extractor is a central component of the KB compilation pipeline and integrates with several other parts of the system:
- **Ingester**: It consumes `IngestedContent` objects produced by the Ingester subsystem.
- **Knowledge Synthesizer**: It produces a `CompilationPlan` which serves as the primary input for the Knowledge Synthesizer.
- **Ontology**: It relies on the `KBOntology` and `ConceptRegistry` for looking up existing entities, validating types, and scanning for mentions.
- **LLM Providers**: It is decoupled from any specific LLM implementation via the `GenerateFn` abstraction, allowing it to work with any YAAF-compatible model adapter.

## Key APIs

### `ConceptExtractor`
The main class of the subsystem. It is instantiated with the KB ontology, registry, and a `GenerateFn`. Its primary method, `buildPlan`, takes an array of `IngestedContent` and returns the `CompilationPlan` [Source 1].

```typescript
// Example Usage
const extractor = new ConceptExtractor(ontology, registry, generateFn);
const plan = await extractor.buildPlan(ingestedContents);
```

### `GenerateFn`
A type definition for a simple asynchronous function that calls an LLM. It accepts a [System Prompt](../concepts/system-prompt.md) and a user prompt and returns the model's text response as a promise. This abstraction allows the extractor to remain provider-agnostic and enhances testability [Source 1].

```typescript
export type GenerateFn = (systemPrompt: string, userPrompt: string) => Promise<string>;
```

### `makeGenerateFn`
A convenience function that creates a `GenerateFn` from any YAAF-compatible model adapter (e.g., a `BaseLLMAdapter` subclass). This simplifies the process of wiring a specific LLM into the extractor [Source 1].

## Extension Points

The primary extension point for the Concept Extractor is the `GenerateFn` provided to its constructor. By supplying a custom implementation of this function, developers can integrate any LLM, use a different set of model parameters, or even substitute a mocked version for testing purposes without altering the extractor's internal logic [Source 1].
