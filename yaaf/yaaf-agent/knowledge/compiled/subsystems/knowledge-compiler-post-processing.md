## Purpose

The [Knowledge Compiler](./knowledge-compiler.md) Post-Processing subsystem is a crucial step in the YAAF knowledge base compilation pipeline. It operates on the raw markdown articles generated by the [LLM](../concepts/llm.md) authoring step, applying a series of deterministic transforms to prepare them for final validation and deployment. Its primary responsibilities are to resolve internal article links and to ensure articles adhere to size constraints by splitting them into smaller, navigable parts. This subsystem runs after article synthesis and before the [Linter](../concepts/linter.md), ensuring that the linter can validate the final, processed structure [Source 1].

## Architecture

The subsystem's architecture is centered around two distinct transformation phases, orchestrated by the main `postProcessCompiledArticles` function [Source 1].

### B1: [Wikilink Resolution](../concepts/wikilink-resolution.md)

This phase converts placeholder [Wikilinks](../concepts/wikilinks.md) into standard, relative markdown links. The `resolveWikilinks` function is responsible for this transformation. It uses a regular expression to find all instances of `Target` or `display text` syntax within an article's text. For each match, it queries the `ConceptRegistry` to find the corresponding document ID for the `Target`.

- If a match is found, the syntax is replaced with a relative markdown link, e.g., `[display text](../concepts/target-doc-id.md)`.
- If the target cannot be found in the registry, the `Target` syntax is left unchanged. These unresolved links are intended to be flagged as errors by the subsequent [Linting](../concepts/linting.md) step [Source 1].

This process also generates dependency data (`wikilinkDeps`) mapping each article to the set of articles it links to, which is used by the differential build engine. A list of articles with unresolved links (`unresolvedDocIds`) is also produced to help target future compilation runs [when](../apis/when.md) new concepts are added [Source 1].

### B2: [Article Segmentation](../concepts/article-segmentation.md)

This phase addresses the issue of oversized articles. The `segmentOversizedArticles` function scans all compiled articles and estimates their token count. If an article exceeds the configured `tokenBudget`, it is split into multiple smaller articles.

- Splitting occurs at H2 (`##`) section boundaries.
- The original article becomes "Part 1," and subsequent sections are moved into new files named with a suffix (e.g., `docId-part-2.md`, `docId-part-3.md`).
- To maintain readability and navigation, the subsystem automatically inserts navigation links at the bottom of each part, such as `[Next: Section Name →]` and `[← Previous]` [Source 1].

## Integration Points

The Post-Processing subsystem is an integral part of the larger knowledge compiler.

- **Upstream:** It receives its input directly from the [LLM Synthesis](../concepts/llm-synthesis.md) stage, which produces markdown files with unresolved `wikilinks`. It also depends on the `ConceptRegistry` provided by the [Ontology](../concepts/ontology.md) subsystem to resolve these links [Source 1].
- **Downstream:** Its output—a directory of processed, linked, and segmented markdown files—is fed into the linter for validation. The dependency metadata it generates (`wikilinkDeps`, `unresolvedDocIds`) is consumed by the differential build engine to optimize incremental compilations [Source 1].

## Key APIs

- **`postProcessCompiledArticles(compiledDir, registry, options)`**: The main entry point for the subsystem. It orchestrates the entire post-processing [workflow](../concepts/workflow.md) over a directory of compiled articles [Source 1].
- **`resolveWikilinks(markdown, registry, currentDocId)`**: A pure function that takes a markdown string and resolves all `wikilink` syntax within it, returning the transformed markdown and dependency information [Source 1].
- **`segmentOversizedArticles(compiledDir, tokenBudget)`**: An asynchronous function that reads articles from the filesystem, splits any that are too large, and writes the new, smaller articles back to disk with navigation links [Source 1].

## Configuration

The behavior of the subsystem can be controlled via the `PostProcessOptions` object passed to `postProcessCompiledArticles`.

- **`tokenBudget`**: A number specifying the maximum token count for a single article before it is segmented. The default is 15,000 tokens [Source 1].
- **`resolveLinks`**: A boolean flag to enable or disable the wikilink resolution phase. Defaults to `true` [Source 1].
- **`segmentArticles`**: A boolean flag to enable or disable the article segmentation phase. Defaults to `true` [Source 1].
