## Purpose
The Worker Runtime subsystem enables the deployment of YAAF agents to edge computing platforms and serverless environments that implement the standard Web Fetch API. It abstracts the complexities of request routing, Server-Sent Events (SSE) streaming, and Cross-Origin Resource Sharing (CORS) into a single handler compatible with runtimes such as Cloudflare Workers, Vercel Edge Functions, Deno Deploy, and Bun.

## Architecture
The subsystem is built around a factory function, `createWorker`, which wraps a YAAF agent instance into a standard Fetch API handler. Internally, the runtime manages:

- **Request Routing**: Directing incoming traffic to specific agent actions based on the URL path.
- **Stream Management**: Utilizing native `ReadableStream` objects to provide SSE capabilities for real-time agent responses.
- **Lifecycle Hooks**: Executing pre-processing and post-processing logic around the agent's execution cycle.
- **Security**: Providing a standardized interface for request authorization and CORS management.

## Key APIs
The primary interface for this subsystem is the `createWorker` function. It typically requires an agent instance—often wrapped in a streaming utility—to produce a platform-agnostic fetch handler.

### Endpoints
The handler generated by the Worker Runtime provides a standardized set of endpoints:

| Method | Path | Description |
|--------|------|-------------|
| `POST` | `/chat` | Standard request/response interaction (JSON) |
| `POST` | `/chat/stream` | Server-Sent Events (SSE) streaming interaction |
| `GET` | `/health` | Subsystem health check |
| `GET` | `/info` | Retrieval of agent metadata |
| `OPTIONS` | `*` | Automatic handling of CORS preflight requests |

## Configuration
The runtime is configured via an options object passed to `createWorker`. This allows developers to tune the runtime for specific platform constraints, such as the lower memory or timeout limits typical of edge functions.

| Option | Type | Default | Description |
|--------|------|---------|-------------|
| `name` | `string` | `'yaaf-agent'` | Identifier for the agent instance |
| `cors` | `boolean` | `true` | Whether to enable CORS headers |
| `corsOrigin` | `string` | `'*'` | Allowed origin for CORS requests |
| `maxBodySize` | `number` | `1MB` | Maximum allowed size for the request body |
| `timeout` | `number` | `30000` | Request timeout in milliseconds |
| `beforeRun` | `Function` | — | Hook for pre-processing input or context |
| `afterRun` | `Function` | — | Hook for post-processing or side effects |
| `authorize` | `Function` | — | Custom logic for validating requests |

### Authorization Example
The `authorize` hook allows for the implementation of custom security logic, such as JWT or API key validation, before the agent processes a request:

```typescript
const handler = createWorker(agent, {
  authorize: async (req) => {
    const token = req.headers.get('Authorization')?.replace('Bearer ', '');
    if (!token) return false;

    // Validate JWT, API key, etc.
    return await validateToken(token);
  },
});
```

## Extension Points
The Worker Runtime provides several hooks to extend its behavior without modifying the core agent logic:

- **`beforeRun(input, req)`**: Allows developers to modify the user input or inject context based on request metadata (e.g., geographic headers from Cloudflare).
- **`afterRun(input, response, req)`**: Facilitates post-execution tasks such as logging to external analytics or monitoring services.
- **Platform-Specific Integration**: Because the output is a standard fetch handler, it can be integrated into various platform entry points:
    - **Cloudflare Workers**: Exported as the `fetch` handler in a module worker.
    - **Vercel Edge**: Returned from a standard Edge Function route.
    - **Deno**: Passed directly to `Deno.serve()`.

### SSE Streaming
The runtime leverages native `ReadableStream` support to handle streaming. When a client requests the `/chat/stream` endpoint, the runtime maintains a persistent connection and pushes data chunks as they are generated by the underlying LLM provider.
