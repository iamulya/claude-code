## Purpose

The [Linter](../concepts/linter.md) Fixer subsystem automatically applies corrections for a specific set of auto-fixable lint issues identified in compiled knowledge base articles [Source 1]. Its purpose is to reduce manual maintenance by programmatically resolving common, well-defined errors such as non-canonical [Wikilinks](../concepts/wikilinks.md), unlinked term mentions, and missing [Frontmatter](../concepts/frontmatter.md) fields [Source 1].

## Architecture

The fixer is designed to be conservative and safe in its operations. It processes a list of lint issues, applying fixes incrementally. Key architectural principles include [Source 1]:

*   **Incremental Application**: Changes are applied for one issue at a time, building up the final state of the file's content in [Memory](../concepts/memory.md).
*   **File-based Grouping**: All changes for a single file are grouped together to ensure that only one write operation is performed per file. This is managed through the use of `atomicWriteFile` to prevent data corruption [Source 1].
*   **Context-aware Skipping**: The fixer checks if the target text for an issue still exists before applying a fix. If a previous fix in the same run has already modified the target text, the subsequent issue is skipped.
*   **Non-destructive Operations**: The subsystem is designed to only add content or rewrite wikilinks. It never removes existing content from an article.

The fixer can handle three types of auto-fixable issues [Source 1]:
*   `NON_CANONICAL_WIKILINK`: Rewrites a wikilink alias to its canonical title (e.g., `alias` becomes `canonical title`).
*   `UNLINKED_MENTION`: Finds the first occurrence of a term that should be linked and wraps it in wikilink syntax.
*   `MISSING_REQUIRED_FIELD`: Injects a required field with a default value into an article's YAML frontmatter.

## Integration Points

The Linter Fixer is designed to be used as part of a larger knowledge base compilation or [Linting](../concepts/linting.md) process. It consumes an array of `LintIssue` objects, which are typically generated by a linter subsystem. It interacts directly with the file system by reading from and writing to the directory containing the compiled articles.

## Key APIs

The primary public API for this subsystem is the `applyFixes` function.

### `applyFixes`

```typescript
export async function applyFixes(
  issues: LintIssue[],
  compiledDir: string,
  dryRun: boolean = false,
): Promise<AutoFixResult>
```

This function takes a list of lint issues and attempts to apply fixes to the corresponding files within the specified compiled directory [Source 1].

*   **`issues`**: An array of `LintIssue` objects to be fixed.
*   **`compiledDir`**: The path to the directory containing the compiled article files.
*   **`dryRun`**: If `true`, the function will calculate the changes but will not write them to disk. This allows for a preview of the potential fixes.
*   **Returns**: A promise that resolves to an `AutoFixResult` object, which contains details about the applied and skipped fixes.
