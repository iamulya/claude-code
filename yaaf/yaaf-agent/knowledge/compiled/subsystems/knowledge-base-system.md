## Purpose
The Knowledge Base (KB) System provides a "compile-time" alternative to traditional Retrieval-Augmented Generation (RAG). Instead of embedding and retrieving document chunks at query time, the system ingests raw source material—such as research papers, web clips, and documentation—and compiles it into a structured, human-readable wiki.

This subsystem ensures that agents have access to high-fidelity, cross-referenced context that is legible to both humans and LLMs. By using a structured ontology and a self-healing linter, the KB System maintains consistency across large volumes of information, bypassing the noise and fragmentation often associated with vector-based retrieval.

## Architecture
The KB System is structured as a multi-stage pipeline that transforms unstructured data into a formal knowledge graph represented as Markdown files.

### Directory Structure
The system operates on a specific directory convention:
*   `ontology.yaml`: The required domain schema defining entity types, relationships, and vocabulary.
*   `raw/`: The input directory containing source material (Markdown, HTML, Code, etc.).
*   `compiled/`: The output directory containing the LLM-authored wiki articles, organized by entity type.
*   `.kb-registry.json`: An auto-maintained index of all compiled articles used for entity resolution.

### Compilation Pipeline
The `KBCompiler` executes the transformation through several distinct stages:

1.  **Ingester**: Normalizes various file formats. It strips frontmatter from Markdown, uses Readability for HTML extraction, and extracts docstrings from source code.
2.  **Concept Extractor**: Performs a two-pass analysis.
    *   **Static Pass**: Scans the source against the known vocabulary and registry to detect existing entities without LLM calls.
    *   **LLM Pass**: Uses a fast "extraction model" to identify new candidate concepts and plan article updates or creations.
3.  **Knowledge Synthesizer**: Uses a high-capability "synthesis model" to author or merge content into coherent articles. It follows the `article_structure` defined in the ontology and injects `[[wikilinks]]` for cross-references.
4.  **Linter**: A self-healing component that checks for broken links, missing required sections, and non-canonical terminology, generating a report for auto-fixable issues.

### Core Components
*   **KBOntology**: The central schema that defines the rules of the knowledge domain, including `entity_types`, `relationship_types`, and `vocabulary`.
*   **ArticlePlan**: A deterministic plan generated by the Extractor that specifies whether to create, update, or skip an article based on source content.
*   **ConceptRegistry**: A live index of all compiled entities, used to resolve mentions to specific `docId` paths.

## Integration Points
The KB System interacts with the broader YAAF framework through:
*   **LLM Providers**: The compiler requires an `extractionModel` and a `synthesisModel`, typically provided via `makeGenerateFn` wrappers around YAAF chat models.
*   **Agent Context**: Compiled articles are served to agents, allowing them to read well-structured documentation as part of their system or tool context.

## Key APIs
*   `KBCompiler.create(options)`: Asynchronously initializes a compiler instance, loading the ontology and registry from disk.
*   `compiler.compile()`: Triggers the full ingestion, extraction, and synthesis pipeline.
*   `makeGenerateFn(model)`: A utility that adapts YAAF chat models for use within the compiler's specialized prompts.

## Configuration
The system is primarily configured via `ontology.yaml`. Key configuration blocks include:

### Entity Types
Defines the categories of knowledge (e.g., `concept`, `tool`, `research_paper`). Each type specifies:
*   **Frontmatter**: A schema-validated block of metadata (e.g., authors, URLs, enums).
*   **Article Structure**: A list of required or optional H2 sections the LLM must write.
*   **Linkable To**: Constraints on which other entity types this article can reference.

### Vocabulary
A mapping of canonical terms to their aliases. This is used by the Extractor to detect mentions and by the Linter to enforce canonical naming.
```yaml
vocabulary:
  "attention mechanism":
    aliases: [attention, self-attention]
    entity_type: concept
    doc_id: concepts/attention-mechanism
```

### Token Budget
Controls the granularity of information served to agents:
*   `text_document_tokens`: Maximum length for a compiled article.
*   `image_tokens`: Budget for vision-related data.
*   `max_images_per_fetch`: Limits the number of visual assets returned during runtime.

## Extension Points
*   **Custom Ontologies**: Developers can define entirely new domains by modifying `ontology.yaml`.
*   **Relationship Types**: Users can define named, reciprocal relationships (e.g., `IMPLEMENTS` / `IMPLEMENTED_BY`) which the linter uses to ensure bidirectional linking.
*   **Model Assignment**: Different LLMs can be assigned to specific roles (Extraction, Synthesis, Analysis, or Vision) to optimize for cost and quality.
