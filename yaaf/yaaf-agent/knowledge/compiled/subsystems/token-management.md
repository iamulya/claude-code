## Purpose

The Token Management subsystem provides [Utilities](./utilities.md) for estimating the token count of text and structured messages without requiring an API call to a specific [LLM](../concepts/llm.md) provider's tokenizer. Its primary purpose is to enable efficient management of an LLM's limited [Context Window](../concepts/context-window.md). By providing fast, conservative token estimates, this subsystem helps prevent [Context Overflow](../concepts/context-overflow.md) errors, manage API costs, and support higher-level [Context Management](./context-management.md) strategies [Source 3].

## Architecture

The core of this subsystem is a set of utility functions located in `src/utils/tokens.ts`. It does not rely on external tokenizer libraries, instead using a character-ratio heuristic for performance and provider-agnosticism [Source 3].

The fundamental estimation logic in `estimateTokens` is based on the following heuristic:
*   Approximately 4 characters per token for English text and source code.
*   Approximately 1.5 characters per token for text with a high concentration (>30%) of CJK (Chinese, Japanese, Korean) characters.

This heuristic is designed to be intentionally conservative, overestimating token counts by 5-15% to create a safety margin against context overflow [Source 3].

Building upon this base function, the subsystem provides higher-level estimators like `estimateMessageTokens` and `estimateConversationTokens`. These functions are aware of the structured format of LLM messages and account for the additional tokens generated by JSON syntax and object overhead in tool-use scenarios [Source 3]. An optimized `exceedsBudget` function is also available for situations where only a boolean check against a token limit is needed, which is more efficient than a full count [Source 3].

## Integration Points

The Token Management subsystem is a foundational utility used by several other parts of the YAAF framework.

*   **[LLM Adapters](./llm-adapters.md)**: The `BaseLLMAdapter` class, which serves as the foundation for all model-specific adapters, incorporates the `estimateTokens` utility to provide a shared, consistent [Token Estimation](../concepts/token-estimation.md) implementation for all models [Source 2].
*   **Context Management**: Features like "[History Snipping](../concepts/history-snipping.md)" rely heavily on this subsystem. The `snipHistory` function uses `estimateTokens` to identify [tool results](../concepts/tool-results.md) that are large enough to be considered for removal and to report the number of `tokensFreed` by the optimization process [Source 1]. This demonstrates its role in pre-compaction passes that reduce context size before more expensive operations.

## Key APIs

The public API of this subsystem consists of several exported functions from `src/utils/tokens.ts`.

*   `estimateTokens(text: string): number`: The core function that estimates the token count for a given string using the character-ratio heuristic [Source 3].
*   `estimateMessageTokens(message: object): number`: Calculates the estimated token count for a single, structured message object, including overhead for its structure [Source 3].
*   `estimateConversationTokens(messages: Array<object>): number`: Aggregates the token count for an entire array of message objects [Source 3].
*   `exceedsBudget(text: string, budgetTokens: number): boolean`: An efficient utility that returns `true` if a string's estimated token count exceeds a specified budget, and `false` otherwise [Source 3].

## Configuration

The core [Token Estimation Heuristic](../concepts/token-estimation-heuristic.md) is not directly configurable. However, subsystems that consume these utilities often expose configuration options based on token counts.

For example, the History Snipping feature can be configured with a `minSnipTokens` parameter, which uses the output of `estimateTokens` to determine the minimum size of a tool result to be eligible for snipping [Source 1].

```typescript
// Example of configuring a feature that uses token estimation
const result = snipHistory(messages, { minSnipTokens: 100 });
```
