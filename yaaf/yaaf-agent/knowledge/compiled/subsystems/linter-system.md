## Purpose

The Linter System is responsible for maintaining the quality, consistency, and integrity of the YAAF [Knowledge Base](./knowledge-base.md). It provides a framework for defining and executing rules that analyze compiled articles, identify potential issues, and report them. These issues can range from broken wikilinks and formatting errors to missing metadata. The system also provides data structures to support automatic fixing of certain classes of problems.

## Architecture

The Linter System is composed of several key parts that work together to analyze the [Knowledge Base](./knowledge-base.md). The primary files suggest a separation of concerns between the public API, data type definitions, and article processing [Source 1].

- **`index.ts`**: Serves as the main entry point or "barrel" file for the subsystem. It aggregates and exports the public API, making types and functions available to other parts of the YAAF framework [Source 1].
- **`types.js`**: This file defines the core data structures used throughout the linting process. It includes types for representing individual issues ([LintIssue](../apis/lint-issue.md)), the severity of an issue ([LintSeverity](../apis/lint-severity.md)), and the final aggregated report ([LintReport](../apis/lint-report.md)) [Source 1]. It also defines structures related to automatic fixing, such as [AutoFixResult](../apis/auto-fix-result.md) [Source 1].
- **`reader.js`**: This component appears to be responsible for reading and parsing compiled knowledge base articles into a standardized format, [ParsedCompiledArticle](../apis/parsed-compiled-article.md), which the linter rules can then operate on [Source 1].

The overall architecture involves reading articles, applying a set of validation rules to each one, and collecting any discovered issues into a comprehensive [LintReport](../apis/lint-report.md).

## Integration Points

The Linter System primarily integrates with other parts of the knowledge compilation pipeline:

- **[Knowledge Base Compiler](./knowledge-base-compiler.md)**: The linter is likely invoked as a post-compilation step to validate the generated articles before they are considered final.
- **[CLI](./cli.md)**: The YAAF [CLI](./cli.md) likely provides commands to run the linter on-demand, allowing developers to check the health of the [Knowledge Base](./knowledge-base.md) manually.
- **[Linter Fixer](./linter-fixer.md)**: A dedicated fixer subsystem would consume the [LintReport](../apis/lint-report.md) generated by the linter to apply automatic corrections based on the [AutoFixResult](../apis/auto-fix-result.md) data.

## Key APIs

The public API of the Linter System is primarily composed of data structures that model the linting process and its outcomes [Source 1].

- **[LintIssue](../apis/lint-issue.md)**: Represents a single validation error or warning found in an article.
- **[LintReport](../apis/lint-report.md)**: A comprehensive report containing all [LintIssue](../apis/lint-issue.md) instances found during a linting run.
- **[LintOptions](../apis/lint-options.md)**: A configuration object that specifies how a linting run should be performed.
- **[LintCode](../apis/lint-code.md)**: An enumeration of unique identifiers for each type of linting rule (e.g., `BROKEN_LINK`, `MISSING_FRONTMATTER`).
- **[LintSeverity](../apis/lint-severity.md)**: Defines the severity level of an issue, such as `ERROR` or `WARNING`.
- **[AutoFixResult](../apis/auto-fix-result.md)**: Describes the outcome of an attempt to automatically fix one or more issues.
- **[FixedIssue](../apis/fixed-issue.md)**: Represents a [LintIssue](../apis/lint-issue.md) that has been successfully resolved by an auto-fixer.
- **[ParsedCompiledArticle](../apis/parsed-compiled-article.md)**: The in-memory representation of a compiled article that the linter's rules analyze.
- **[LinkGraph](../apis/link-graph.md)**: A data structure likely used to represent the network of links between articles, enabling efficient detection of broken or orphaned links.

## Configuration

The behavior of the Linter System can be customized through the [LintOptions](../apis/lint-options.md) API [Source 1]. This allows developers to enable or disable specific rules, change the severity of issues, or provide other parameters that control the validation process.
