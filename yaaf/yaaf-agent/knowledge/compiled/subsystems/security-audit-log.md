## Purpose
The Security Audit Log subsystem provides a centralized mechanism for collecting and persisting security-related events generated by YAAF security middleware. It is designed to create a structured, queryable audit trail essential for regulatory compliance, forensic analysis, and threat intelligence.

The subsystem addresses the need for visibility into the security posture of LLM-powered agents by tracking events such as prompt injections, PII detection, and tool access violations.

## Architecture
The subsystem is built around an append-only structured event log. It categorizes events by severity and type, allowing for granular filtering and analysis.

### Core Components
- **SecurityAuditLog**: The primary class responsible for managing the lifecycle of audit entries, including storage, eviction, and statistics generation.
- **AuditEntry**: The fundamental data structure representing a security event. It includes metadata such as unique IDs, timestamps, severity levels, and correlation IDs (User and Session).
- **Severity Levels**: Events are classified into three levels: `info`, `warning`, and `critical`.
- **Event Categories**: The system tracks specific security domains, including:
    - `prompt_injection`
    - `pii_detected` / `pii_redacted`
    - `trust_violation`
    - `grounding_failed`
    - `tool_blocked`
    - `canary_triggered`

### Data Retention
The log implements a rolling retention policy. By default, it retains the last 10,000 entries in memory, evicting the oldest entries when the limit is reached.

## Integration Points
The Security Audit Log acts as a sink for security events produced by other parts of the framework:
- **Security Middleware**: Feeds events into the log during input/output processing.
- **IAM (Identity and Access Management)**: Provides `userId` context for event correlation.
- **External Systems**: Through the `onEntry` callback, the subsystem can forward events in real-time to external Security Information and Event Management (SIEM) systems, such as CloudWatch or Datadog.

## Key APIs
The subsystem exposes a factory function and a management class.

### securityAuditLog()
A factory function used to initialize the audit log with a specific configuration.

### SecurityAuditLog Class
The main interface for interacting with the audit trail. It provides methods for:
- Appending new entries.
- Querying the log.
- Generating `AuditStats`, which include summaries by severity, category, source, and top users.
- Exporting data to JSON or NDJSON formats.

## Configuration
The subsystem is configured via the `AuditLogConfig` object:

| Property | Type | Description |
| :--- | :--- | :--- |
| `maxEntries` | `number` | Maximum entries to retain (Default: 10,000). |
| `minSeverity` | `AuditSeverity` | Minimum severity level to record (Default: `info`). |
| `onEntry` | `Function` | Real-time callback for event forwarding. |
| `sessionId` | `string` | Optional ID for correlating events to a specific session. |

```typescript
const config: AuditLogConfig = {
  maxEntries: 5000,
  minSeverity: 'warning',
  onEntry: (entry) => {
    console.log(`Security Alert: ${entry.summary}`);
  }
};
```

## Extension Points
Developers can extend the functionality of the Security Audit Log through:
- **Custom Categories**: The `AuditCategory` type supports a `custom` value for user-defined security events.
- **Real-time Forwarding**: The `onEntry` hook allows developers to implement custom persistence layers or alerting logic (e.g., sending a Slack notification on `critical` events).
- **Statistics**: The `AuditStats` structure provides a foundation for building custom security dashboards or threat intelligence reports.
