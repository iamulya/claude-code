## Overview

The `KB[[[[[[[[Linter]]]]]]]]` class is the primary orchestrator for validating a compiled YAAF Knowledge Base (KB) [Source 1]. It scans the compiled articles, runs a series of static checks in parallel, and aggregates the findings into a structured `LintReport`.

This class is a key component of the "self-healing" process for the KB, which involves three main steps [Source 1]:
1. Scan the compiled KB for issues.
2. Automatically fix any issues that can be resolved programmatically.
3. Output a human-readable report of any remaining issues that require manual intervention.

In addition to identifying issues, `KBLinter` can also apply automatic fixes based on a generated report, making it a powerful tool for maintaining KB quality and consistency [Source 1].

## Constructor

The `KBLinter` is instantiated with the knowledge base [Ontology](../concepts/ontology.md), [Concept Registry](../subsystems/concept-registry.md), and the path to the compiled knowledge base directory [Source 1].

```typescript
import type { KBOntology, ConceptRegistry } from "../../ontology/index.js";

export class KBLinter {
  constructor(
    ontology: KBOntology,
    registry: ConceptRegistry,
    compiledDir: string
  );
}
```

**Parameters:**

*   `ontology`: `KBOntology` - The parsed ontology defining the structure and types of the knowledge base.
*   `registry`: `ConceptRegistry` - The registry containing all known concepts.
*   `compiledDir`: `string` - The file system path to the directory containing the compiled knowledge base articles.

## Methods & Properties

The `KBLinter` class exposes methods for [Linting](../concepts/linting.md) the knowledge base and applying fixes.

### lint()

Scans the compiled KB, runs all static checks, and returns a report of any issues found [Source 1].

```typescript
import type { LintReport } from "./types.js";

async lint(): Promise<LintReport>;
```

**Returns:**

*   `Promise<LintReport>`: A promise that resolves to a `LintReport` object containing a list of all detected `LintIssue` items.

### fix()

Takes a `LintReport` and attempts to apply all available automatic fixes for the issues it contains [Source 1].

```typescript
import type { LintReport, AutoFixResult } from "./types.js";

async fix(report: LintReport): Promise<AutoFixResult>;
```

**Parameters:**

*   `report`: `LintReport` - The report generated by a prior call to `lint()`.

**Returns:**

*   `Promise<AutoFixResult>`: A promise that resolves to an `AutoFixResult` object, which summarizes the files that were changed and the fixes that were applied.

## Examples

The following example demonstrates the standard usage pattern: first, create a Linter instance, then generate a report, and finally, apply automatic fixes based on that report [Source 1].

```typescript
import { KBLinter } from 'yaaf';
import { ontology, registry } from './my-kb-config';

const compiledDir = './dist/kb';

// 1. Instantiate the linter
const linter = new KBLinter(ontology, registry, compiledDir);

// 2. Scan for issues and generate a report
const report = await linter.lint();
// report now contains a LintReport with all issues

console.log(`Found ${report.issues.length} issues.`);

// 3. Apply auto-fixes based on the report
const fixes = await linter.fix(report);
// fixes now contains an AutoFixResult with what was changed

console.log(`Applied ${fixes.applied.length} fixes across ${fixes.filesChanged.length} files.`);
```
