## Overview

The `ProposalFile` type represents the structure of a file containing a list of proposed changes to the knowledge base [Ontology](../concepts/ontology.md) [Source 1]. It is used within the [Knowledge Base Compiler](../subsystems/knowledge-base-compiler.md) subsystem to validate suggestions for evolving the ontology, which may be generated by an [LLM](../concepts/llm.md) or a user.

Each proposal in the file specifies a kind of change (e.g., adding an [Entity Type](../concepts/entity-type.md), a [Vocabulary](../concepts/vocabulary.md) term, a relationship, or a field), along with a confidence score and a justification [Source 1]. This structured format allows for the automated processing and review of suggestions to extend the knowledge base's schema.

The type is derived from the Zod schema `ProposalFileSchema`, which ensures that any proposal data conforms to the expected contract before being processed by the compiler [Source 1].

### Proposal Object Properties

Each object in the `proposals` array has the following properties:

| Property     | Type                                                                      | Description                                                                                             |
|--------------|---------------------------------------------------------------------------|---------------------------------------------------------------------------------------------------------|
| `kind`       | `"add_entity_type" \| "add_vocabulary" \| "add_relationship" \| "add_field"` | The type of ontology change being proposed.                                                             |
| `confidence` | `number`                                                                  | A score between 0.0 and 1.0 indicating the confidence in the proposal's validity.                       |
| `reason`     | `string`                                                                  | A human-readable justification for the proposed change.                                                 |
| `entityType` | `string` (optional)                                                       | The target entity type for the proposal.                                                                |
| `term`       | `string` (optional)                                                       | The vocabulary term being proposed (e.g., for `add_vocabulary`).                                        |
| `from`       | `string` (optional)                                                       | The source entity type for a relationship proposal.                                                     |
| `to`         | `string` (optional)                                                       | The destination entity type for a relationship proposal.                                                |
| `details`    | `Record<string, unknown>` (optional)                                      | A key-value map for any additional, unstructured details relevant to the proposal.                      |

## Examples

A typical `ProposalFile` object might contain several suggestions for improving the ontology.

```typescript
import { ProposalFile } from 'yaaf';

const ontologyProposals: ProposalFile = {
  proposals: [
    {
      kind: "add_entity_type",
      confidence: 0.95,
      reason: "The source material frequently discusses API endpoints as distinct concepts.",
      entityType: "api_endpoint",
      details: {
        suggestedFields: ["path", "method", "description"]
      }
    },
    {
      kind: "add_vocabulary",
      confidence: 0.88,
      reason: "The term 'RESTful Service' is used interchangeably with 'api'.",
      entityType: "api",
      term: "RESTful Service"
    },
    {
      kind: "add_relationship",
      confidence: 0.92,
      reason: "Agents are consistently described as using tools to perform actions.",
      from: "agent",
      to: "tool",
      details: {
        relationshipName: "uses"
      }
    }
  ]
};

// This object can now be validated against ProposalFileSchema
// before being processed by the knowledge compiler.
```
