## Overview

`InProcessIPCConfig` is the configuration interface for the [InProcessIPCPlugin](../plugins/in-process-ipc-plugin.md). Its primary purpose is to allow developers to replace the default in-process messaging system with a custom, distributed backend for inter-process communication (IPC) [Source 1].

By default, the [InProcessIPCPlugin](../plugins/in-process-ipc-plugin.md) uses an in-memory transport (equivalent to `InMemoryIPCBackend`), which means messages are only visible within a single Node.js process. In multi-replica or multi-pod deployments, this prevents agents from communicating with each other across process boundaries [Source 1].

To enable cross-replica communication, a user can provide an implementation of the [DistributedIPCBackend](./distributed-ipc-backend.md) interface via the `backend` property of this configuration object. This delegates message transport to a shared backing store, such as Redis, allowing all connected agent replicas to send and receive messages from shared mailboxes [Source 1].

## Signature

The `InProcessIPCConfig` interface allows for the specification of a distributed backend.

```typescript
import type { DistributedIPCBackend } from 'yaaf';

export interface InProcessIPCConfig {
  /**
   * An optional distributed backend for cross-replica message transport.
   * When provided, the InProcessIPCPlugin delegates send, subscribe, and
   * read operations to this backend. If not provided, the plugin defaults
   * to an in-process EventEmitter, limiting IPC to the current Node.js process.
   */
  backend?: DistributedIPCBackend;
}
```

## Examples

The following example demonstrates how to configure the [InProcessIPCPlugin](../plugins/in-process-ipc-plugin.md) with a Redis-based distributed backend. This setup enables agents running in different processes or on different machines to communicate, provided they connect to the same Redis instance [Source 1].

```typescript
import { Agent, InProcessIPCPlugin } from 'yaaf';
import { createClient } from 'redis';
import type { DistributedIPCBackend, IPCMessage, InProcessIPCConfig } from 'yaaf';

// 1. Set up Redis clients for publishing and subscribing
const pub = createClient();
const sub = pub.duplicate();
await pub.connect();
await sub.connect();

// 2. Create a DistributedIPCBackend implementation using Redis
const redisBackend: DistributedIPCBackend = {
  async publish(mailbox, message) {
    const key = `ipc:inbox:${mailbox}`;
    // Use LPUSH to add to a list, ensuring FIFO order for draining
    await pub.lPush(key, JSON.stringify(message));
    // Use a separate pub/sub channel for real-time notifications
    await pub.publish(`ipc:ch:${mailbox}`, JSON.stringify(message));
    return pub.lLen(key);
  },
  subscribe(mailbox, handler) {
    const channel = `ipc:ch:${mailbox}`;
    sub.subscribe(channel, (msg) => handler(JSON.parse(msg)));
    return () => sub.unsubscribe(channel);
  },
  async drain(mailbox) {
    const key = `ipc:inbox:${mailbox}`;
    // Atomically get all items and delete the list
    const items = await pub.lRange(key, 0, -1);
    await pub.del(key);
    return items.reverse().map(i => JSON.parse(i) as IPCMessage); // lRange is LIFO, reverse for FIFO
  },
};

// 3. Create the configuration object
const ipcConfig: InProcessIPCConfig = {
  backend: redisBackend,
};

// 4. Initialize an agent with the plugin and its configuration
const agent = new Agent({
  plugins: [
    new InProcessIPCPlugin(ipcConfig),
  ],
  // ... other agent configuration
});

// Now, agent.ipc.send() will use the Redis backend to communicate
// with other agents configured with the same backend.
```
