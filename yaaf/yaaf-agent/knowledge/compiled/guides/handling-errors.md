## Overview
YAAF provides a structured, typed error hierarchy designed to simplify error handling across different LLM providers and framework subsystems. All framework-specific errors extend a base `YAAFError` class, allowing developers to implement unified catch blocks while still accessing granular diagnostic information like machine-readable error codes and retry logic.

## Prerequisites
- A YAAF project configured with a model provider.
- Basic understanding of TypeScript error handling (try/catch blocks).

## Step-by-Step

### 1. Catching Base YAAF Errors
To handle any error generated by the framework (whether from a model provider, a tool execution, or a sandbox violation), catch the `YAAFError` base class. This class provides the `code` and `retryable` properties.

```typescript
import { YAAFError } from 'yaaf';

try {
  const response = await agent.run(task);
} catch (err) {
  if (err instanceof YAAFError) {
    // Handle framework-specific error
    console.error(`Error Code: ${err.code}`);
    console.error(`Is Retryable: ${err.retryable}`);
  }
}
```

### 2. Handling Rate Limits
Specific subclasses like `RateLimitError` provide additional metadata. When a provider returns a rate limit error, YAAF attempts to parse the `Retry-After` header into milliseconds.

```typescript
import { YAAFError, RateLimitError } from 'yaaf';

try {
  await model.complete(params);
} catch (err) {
  if (err instanceof RateLimitError) {
    console.log(`Rate limited. Retry after ${err.retryAfterMs}ms`);
    // Implement custom backoff logic here
  } else if (err instanceof YAAFError) {
    console.log(`YAAF Error [${err.code}]: ${err.message}`);
  }
}
```

### 3. Using Error Codes for Logic
The `code` property allows for machine-readable branching logic without relying on string matching against error messages.

```typescript
try {
  await tool.execute(input);
} catch (err) {
  if (err instanceof YAAFError) {
    switch (err.code) {
      case 'CONTEXT_OVERFLOW':
        // Trigger a compaction or summarization strategy
        break;
      case 'TOOL_EXECUTION_ERROR':
        // Notify the user or attempt a fallback tool
        break;
      case 'AUTH_ERROR':
        // Refresh API keys or credentials
        break;
      default:
        console.error('Unhandled YAAF error', err);
    }
  }
}
```

### 4. Classifying Custom API Errors
If you are building a custom provider adapter, use the `classifyAPIError` utility to convert raw HTTP responses into the appropriate YAAF error types.

```typescript
import { classifyAPIError } from 'yaaf';

const response = await fetch(url, options);

if (!response.ok) {
  const body = await response.text();
  throw classifyAPIError(
    response.status,
    body,
    'my-custom-provider',
    response.headers
  );
}
```

## Configuration Reference

### Error Codes
The following `ErrorCode` values are used across the framework:

| Code | Description |
| :--- | :--- |
| `RATE_LIMIT` | Provider rate limits exceeded. |
| `OVERLOADED` | Provider service is currently overloaded. |
| `AUTH_ERROR` | Authentication or API key issues. |
| `API_ERROR` | General provider API error. |
| `API_CONNECTION_ERROR` | Network-level connectivity issues. |
| `CONTEXT_OVERFLOW` | Input exceeds the model's context window. |
| `TOOL_EXECUTION_ERROR` | Failure during the execution of a tool. |
| `SANDBOX_VIOLATION` | Code execution exceeded sandbox constraints. |
| `PERMISSION_DENIED` | Insufficient permissions for the requested action. |
| `ABORT` | The request was manually aborted. |
| `RETRY_EXHAUSTED` | Maximum retry attempts reached. |
| `COMPACTION_ERROR` | Failure during context compaction. |
| `UNKNOWN` | Unclassified error. |

## Common Mistakes
1. **Ignoring the `retryable` flag**: Not all errors should be retried. Always check `err.retryable` before initiating a retry loop to avoid infinite loops on `AUTH_ERROR` or `CONTEXT_OVERFLOW`.
2. **String Matching Messages**: Avoid using `err.message.includes('rate limit')`. Use `err instanceof RateLimitError` or `err.code === 'RATE_LIMIT'` for reliable identification.
3. **Missing Provider Context**: When catching errors in multi-provider environments, check the `err.provider` property to identify which specific service failed.
